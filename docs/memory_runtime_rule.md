# 一、文档目的

本文档定义 `evo_skills` 在运行时如何使用、更新和维护长期记忆。

它解决的问题是：

- 什么时候该读取 memory？
- 读哪些 memory？
- 什么时候该写回？
- 写回什么，不写回什么？
- 如何避免 memory 越写越脏？

在当前阶段，`evo_skills` 的最小 memory runtime 支持四类操作：

- 初始化用户 memory
- 按 mode 读取有效 memory context
- 写入 session 摘要
- 将 session 摘要压缩进长期 memory

# 二、memory 的角色定位

在 `evo_skills` 中，memory 不是原始对话存档，而是：

- 用户与某个 child skill 长期互动后的稳定摘要
- 用户目标、约束、偏好、卡点和成长变化的结构化记录

因此，memory 应服务于：

- 更有针对性的 `teach`
- 更贴合现实场景的 `coach`
- 更有历史连续性的 `reflect`
- 更安全一致的 `execute`

# 三、运行时何时读取 memory

## 1. teach 模式

可读 memory，但只需轻量读取：

- 用户容易混淆的概念
- 偏好的解释方式

## 2. coach 模式

通常应读取 memory：

- 用户目标
- 用户约束
- 反复出现的卡点
- 近期有效策略

## 3. reflect 模式

必须读取 memory：

- `progress`
- 长期变化摘要
- 反复出现的行为模式

## 4. execute 模式

如果 skill 支持 execute，建议读取：

- 用户偏好的执行风格
- 历史上有效的安全默认项
- 需要避免的高风险偏好

# 四、运行时优先读取哪些层

推荐按以下顺序读取：

## 1. `profile`

读取用户背景、目标、偏好、约束。

## 2. `progress`

读取阶段性总结和长期变化。

## 3. 近期摘要

如果有最近一次的高质量总结，应优先于陈旧 session 片段。

## 4. session 记录

只有在上层摘要不足时，才参考更细粒度的 session 信息。

原则上：

- 优先摘要
- 不优先原始流水记录

# 五、何时写回 memory

不是每轮对话都必须写 memory。

推荐在以下情况写回：

## 1. 用户目标发生明确变化

例如：

- 新目标建立
- 优先级改变
- 约束改变

## 2. 发现了稳定、可长期保留的模式

例如：

- 重复犯同一种判断错误
- 一直偏好某种解释方式
- 某种策略反复有效

## 3. 形成了新的阶段总结

例如：

- 一轮复盘后形成清晰进展结论
- 一段时间训练后出现稳定变化

## 4. execute 过程中形成了稳定偏好

例如：

- 用户总是更偏向先看计划再执行
- 用户对某些高风险动作一贯要求确认

# 六、什么不应该写入 memory

以下内容默认不应写入长期 memory：

- 密钥
- 凭证
- 敏感隐私原文
- 完整逐字对话
- 大段日志
- 一次性情绪化表达且无长期价值的内容

原则是：

- 写摘要
- 不写流水账
- 写稳定事实
- 不写瞬时噪音

# 七、推荐的写回结构

当写回 memory 时，建议尽量压缩到以下几类：

## 1. `profile`

适合写入：

- 长期目标
- 角色背景
- 偏好
- 约束

## 2. `progress`

适合写入：

- 阶段性变化
- 新出现的稳定模式
- 哪些策略开始有效 / 失效

## 3. `sessions`

适合短期留存：

- 重要事件摘要
- 需要下次延续的上下文

但不应长期无限增长，应周期性压缩。

# 八、memory 压缩原则

长期来看，memory 必须压缩，否则会越来越脏。

推荐原则：

1. session 先记短摘要
2. 多轮后压缩成阶段性 summary
3. 再把稳定信息沉淀进 profile / progress

也就是说：

- session -> periodic summary -> durable memory

# 九、写回失败信号

如果出现以下情况，说明 memory 写回策略有问题：

- 每轮都写，导致 memory 爆炸
- 写的都是逐字对话而不是摘要
- 重要目标和约束被淹没在杂讯里
- reflect 模式读不出清晰成长线索

# 十、结论

memory runtime 的核心不是“多记”，而是“记得准、记得稳、记得能用”。

在 `evo_skills` 中，长期记忆应始终服务于：

- 更贴合用户的解释
- 更精准的教练分析
- 更清晰的成长复盘
- 更一致、更安全的执行偏好

最小 runtime 闭环建议按以下顺序执行：

1. 从 shared templates 初始化用户目录
2. 在交互前按 mode 读取 memory context
3. 在交互后写入简短 session summary
4. 定期把 session 摘要压缩进 `profile.md` 与 `progress.md`
