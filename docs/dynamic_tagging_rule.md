# 一、文档目的

本文档定义 `evo_skills` 的 **动态标签系统规则**。

它的目标不是预先设计一棵固定分类树，而是回答下面这些问题：

- 标签应该采用什么格式？
- 什么时候应复用已有标签？
- 什么时候应新建标签？
- 什么时候应合并标签？
- 什么时候应拆分标签？
- routing 时如何利用标签实现渐进式披露？

因此，本文档是一个**动态演化规则文档**，而不是静态标签词表。

# 二、设计原则

## 1. 不预设固定分类体系

`evo_skills` 不应预先硬编码一个固定学科树，例如“心理学 / 编程 / 经济学 / 商业”等，并强迫所有 skill 套进去。

原因：

- 后续 skill 主题会持续扩展
- 不同领域会产生交叉主题
- 预先设计好的树很容易过早僵化

## 2. 只约束标签格式与判断规则

系统应固定的是：

- 标签的命名格式
- 标签的使用规则
- 标签的演化规则

而不是固定标签内容。

## 3. 优先复用，谨慎新建

标签系统如果没有复用优先机制，很快就会变成同义词堆积的混乱标签池。

因此默认规则应是：

- 先复用
- 再新建
- 必要时合并
- 明显过大时拆分

## 4. 标签服务于 routing，而不是替代 routing

标签的价值在于：

- 帮助后台缩小候选 skill 集合
- 在候选过多时给用户做渐进式披露

标签不是唯一决策依据，后续仍要结合：

- 用户意图类型
- mode 兼容性
- memory 关联性
- execute 风险

# 三、标签格式规则

## 1. 标签逻辑标识：`tag_id`

推荐格式：

- 使用小写英文
- 使用 `.` 表示层级
- 不要求固定层级深度

示例：

- `psychology`
- `psychology.decision-making`
- `programming`
- `programming.python`
- `economics.behavioral`
- `learning.meta-learning`

## 2. 标签显示名：`display_name`

用于面向用户或说明文档展示。

示例：

- `心理学`
- `心理学 / 决策`
- `编程 / Python`
- `经济学 / 行为经济学`

## 3. 标签别名：`aliases`

用于处理：

- 同义词
- 中英混合表达
- 用户可能使用的常见叫法

示例：

- `decision-making`
- `决策心理`
- `行为经济`
- `python开发`

## 4. 标签状态：`status`

推荐最少支持：

- `active`
- `merged`
- `deprecated`

原因：

- 标签是动态演化的，不能假设永远稳定不变。

# 四、skill 挂标签的规则

每个 child skill 不应只挂一个粗粒度标签。

推荐至少考虑以下三类标签角色：

## 1. 领域标签（domain-like）

表示它属于哪个大领域。

例如：

- `psychology`
- `programming`
- `economics`
- `learning`

## 2. 主题标签（topic-like）

表示它具体关注的问题。

例如：

- `decision-making`
- `refactoring`
- `behavior-change`
- `risk-analysis`

## 3. 功能标签（function-like，可选）

表示它主要是用来做什么的。

例如：

- `teaching`
- `coaching`
- `execution`
- `reference`

系统可以把这些都统一存成 `tags`，不一定必须拆成多个字段，但在语义上应保留这三种角色意识。

# 五、何时复用已有标签

默认应优先复用已有标签，而不是新建。

应复用已有标签的情况包括：

1. 新 skill 的主题和已有标签高度重合
2. 新标签只是已有标签的同义表达
3. 新标签不会显著提高 routing 区分度
4. 新标签只是语言层面更好听，但语义并没有新增边界

例如：

- 已有：`psychology.decision-making`
- 不应再新建：
  - `decision-psychology`
  - `psych.decisions`
  - `心理决策`

这些应进入 alias，而不是成为新标签。

# 六、何时新建标签

只有在下面条件同时基本满足时，才应新建标签：

1. 新 skill 的主题无法被现有标签充分覆盖
2. 该主题具有长期复用价值，不是一时描述
3. 预计未来可能不止一个 skill 会挂这个标签
4. 标签名字可以稳定解释，而不是临时措辞

不应新建标签的情况：

- 只是某个 skill 的局部细节
- 只是当前文章中的单一例子
- 只是过于空泛的概念，例如：
  - `knowledge`
  - `growth`
  - `thinking`
- 只是已有标签的表述变体

# 七、何时合并标签

动态标签系统必须允许标签合并，否则会逐渐碎片化。

应考虑合并的情况包括：

1. 两个标签长期挂载的 skill 高度重叠
2. 两个标签本质上只是同义词或近义词
3. 两个标签的边界已经无法稳定解释
4. 两个标签在 routing 中没有带来实际区分价值

合并后的建议处理方式：

- 保留一个主标签为 `active`
- 另一个标签改为 `merged`
- 在别名或映射表中保留旧标签指向新标签

# 八、何时拆分标签

当某个标签过大、过泛时，应允许拆分。

应考虑拆分的情况包括：

1. 单个标签下 skill 数量过多
2. 标签下已经形成稳定子主题
3. 该标签在 routing 中已经失去收缩能力
4. 用户在这个标签内部仍频繁需要进一步筛选

例如：

- `programming` 下 skill 过多
- 可进一步拆分为：
  - `programming.python`
  - `programming.javascript`
  - `programming.refactoring`

# 九、routing 中如何使用标签

标签的主要价值不是展示，而是先在后台做候选收缩。

推荐使用方式如下：

## 1. 第一层：后台先按标签收缩候选集

当用户提问时：

- 先做意图识别
- 再用标签匹配缩小 skill 集合

## 2. 第二层：候选过多时，再向用户展示标签层

如果后台筛完之后候选仍然很多，不要立刻把完整 skill 总表摊给用户。

应优先向用户展示标签层，例如：

- 这更像“心理学 / 决策”还是“经济学 / 风险判断”？

## 3. 第三层：在标签内再选具体 skill

这样就能实现渐进式披露：

- 先缩小到类
- 再缩小到具体 skill

# 十、与现有 routing 的关系

标签不能替代现有 runtime routing 规则，而应作为前置筛选层。

完整 routing 顺序建议为：

1. 用户意图识别
2. 标签过滤候选 skill
3. mode 兼容性检查
4. execute 风险检查
5. memory 关联度排序
6. 必要时再向用户展示标签层或 skill 层

# 十一、最低实现建议

即使暂时不修改 registry 结构，后续实现时也至少应为每个 skill 预留：

- `tags`
- 或 `domain_tags` / `topic_tags` / `function_tags`

如果要极简，推荐先统一成：

```json
{
  "skill_id": "coach.clear-thinking.decision-review.v1",
  "tags": [
    "psychology",
    "psychology.decision-making",
    "coaching"
  ]
}
```

后续再视需要拆成多个字段。

# 十二、结论

对 `evo_skills` 而言，最合理的不是固定分类树，而是：

- 使用动态标签系统
- 只固定标签格式与演化规则
- 让当前 agent 在新建、复用、合并、拆分标签时自主判断
- 让标签优先服务于 routing 与渐进式披露

这套机制既能避免前期过度设计，也能避免后期 skill 数量增大后 routing 崩坏。
