# 一、研究目标与结论摘要

本文档用于调研“把文章、书籍、教程、技术文档、笔记、转录等材料蒸馏成可复用 agent skill”时，什么样的蒸馏流程更稳妥、更高质量，以及不同材料类型分别适合哪类蒸馏路径。

本次调研的核心结论如下：

1. 最佳蒸馏流程通常不是“一次性总结”，而是**多阶段管线**：采集、清洗、切分、结构标注、提炼、操作化、验证、迭代。
2. 高质量蒸馏的终点不是“把原文缩短”，而是把内容重写成 **agent 可调用的工作单元**：何时使用、输入是什么、步骤怎么走、输出长什么样、常见错误与边界在哪里。
3. 不同材料适合不同流程。教程与操作指南更适合“操作型 skill”流程，技术文档更适合“参考规范 + 任务包装”流程，文章与书籍更适合“原则 + 决策规则”流程，会议纪要和转录更适合“约束 / 决策 memory”流程。
4. 对 `evo_skills` 而言，最适合的是 **agent-only 的交互式分阶段蒸馏**：先分类，再提炼，再形成结构化 spec，再审查，再 build，而不是从原文直接跳到最终 skill 文件。

# 二、什么是高质量蒸馏流程

高质量蒸馏不应理解为“写一份很短的摘要”，而应理解为：

- 先保留原材料的核心语义和适用边界；
- 再根据 agent 的用途，把这些内容重组成可调用的 skill 结构；
- 最后通过真实小任务或检查表验证这份蒸馏结果能否稳定驱动正确行为。

因此，一个好的蒸馏过程必须同时满足：

- **可触发**：知道何时调用；
- **可执行**：知道如何使用；
- **可验证**：知道如何判断做对了；
- **可复用**：知道哪些是通用规则、哪些只是例子；
- **可追溯**：知道关键判断来自哪里。

# 三、常见高质量蒸馏流程模式

# 一、Capture -> Normalize -> Distill -> Operationalize -> Verify

这是最通用、最稳妥的一条主干流程。

## 1. Capture

收集原材料，保留来源、作者、日期、版本、类型等元信息。

## 2. Normalize

统一格式，去掉明显噪音，把内容切成语义块。

这一步特别适合处理：

- OCR 结果
- 转录文本
- 冗长教程
- 零散笔记

## 3. Distill

提炼以下几类高价值内容：

- 核心概念
- 判断规则
- 操作步骤
- 边界条件
- 常见误用

## 4. Operationalize

把知识重写成 agent 可调用的结构，而不是普通摘要。这个阶段需要回答：

- 什么时候该用这个 skill？
- 输入是什么？
- 步骤有哪些？
- 输出应该长什么样？
- 何时停止？
- 哪些情况要用户确认？

## 5. Verify

拿真实小任务或最小示例试跑，验证 skill 是否真的可用。

这条流程最适合 `evo_skills` 的 agent-only 路径，因为它既保留结构化思考，又允许中途插入用户 review gate。

# 二、Source-Faithful -> Task-Oriented 双层蒸馏

这是一种很适合复杂材料的方式。

第一层是**忠实版提炼**：

- 保留原始作者意图
- 保留专业术语
- 保留限制和例外

第二层是**任务版重构**：

- 围绕 agent 实际要做的动作改写
- 变成触发条件、决策规则、工作步骤、输出模板

这种方法特别适合：

- 长书
- 论文
- 复杂技术标准
- 有很多细节和限定条件的深度文章

它的优势在于：

- 不容易把原始内容过早压扁
- 不容易把 nuanced 观点错误地转成绝对规则

# 三、Progressive Layering 逐层提炼

这类流程强调“分层压缩，而不是一次压到底”。

推荐层次如下：

1. 原文 / 摘录
2. 高价值片段
3. 关键词 / 关键句
4. 规则、步骤、判据
5. skill 草案
6. 经过验证的正式 skill

它特别适合：

- 材料量大
- 需要边读边沉淀
- 不希望一开始就做高成本完整重写

对 `evo_skills` 来说，这也很重要，因为不是所有内容都值得直接生成正式 child skill。有些材料更适合先停留在“distilled article package”阶段。

# 四、先做材料类型判断，再决定蒸馏路线

高质量蒸馏往往不是从“写什么”开始，而是从“这份材料属于哪类内容”开始。

一个非常有效的分类框架是把材料区分为：

- Tutorial
- How-to
- Reference
- Explanation

如果在蒸馏前不先分清内容类型，就很容易出现：

- 把原理解释当成操作步骤
- 把参考文档硬改成 SOP
- 把一次性会议结论误写成通用原则

因此，**先分类，再蒸馏** 应成为 `evo_skills` 的前置动作。

# 四、不同材料类型适合哪种蒸馏流程

# 一、文章 / 博客 / 长文

最适合的蒸馏方式是：

- 提取核心 thesis
- 识别 supporting evidence
- 标出适用场景
- 标出边界和误用
- 最后收敛成原则型或判别型 skill

为什么：

- 文章通常观点密度高，但操作步骤密度低；
- 直接把文章硬转成 SOP 往往会失真。

适合生成：

- 原则型 skill
- 判别型 skill
- 教练型 skill

# 二、书籍

最适合的蒸馏方式是：

- 先做章节地图
- 再抽章节核心模型
- 再压缩出跨章节重复出现的原则
- 最后形成框架型 skill 或长期教练型 skill

为什么：

- 书有大量铺陈、案例和重复论证；
- skill 真正需要的是“模型、触发条件、反模式”。

适合生成：

- 框架型 skill
- 教练型 skill
- 反思型 skill

# 三、教程 / 课程 / 操作指南

最适合的蒸馏方式是：

- 明确前置条件
- 抽取步骤顺序
- 定义每步成功信号
- 补充常见错误与排错分支
- 明确回滚方法

为什么：

- 这类材料天生就接近操作型 skill。

适合生成：

- 操作型 skill
- `execute` 候选 skill

# 四、技术文档 / API 文档 / 官方说明

最适合的蒸馏方式是：

- 先做 reference normalization
- 再做 task wrapper

也就是说，先保留：

- 能力边界
- 参数说明
- 限制条件
- 版本差异

再围绕高频任务写出：

- 常用操作模板
- 输入输出模板
- 常见错误与限制

为什么：

- 官方文档通常完整但不任务友好；
- skill 不应该重写所有 reference，而应该做“任务包装层”。

适合生成：

- 参考型 skill
- 混合型 skill（reference + how-to）

# 五、笔记 / 零散摘录

最适合的蒸馏方式是：

- 去重
- 聚类
- 主题命名
- 抽取重复出现的判断标准
- 最后做轻量 skill 或 memory note

为什么：

- 零散笔记的噪音和上下文残留很多；
- 不先聚类就直接做 skill，质量很难稳定。

适合生成：

- 轻量原则型 skill
- personal memory / project memory

# 六、会议纪要 / 访谈 / 转录

最适合的蒸馏方式是：

- 区分事实与意见
- 区分已决定与未决定
- 提取约束、禁忌、优先级
- 形成项目约束型 skill 或 memory，而不是急着做通用 skill

为什么：

- 转录材料信息多但结构差；
- 它们最大的价值通常是“约束”和“偏好”，不一定是通用知识。

适合生成：

- 约束型 skill
- runtime memory
- project-specific guardrail

# 五、蒸馏结果的评估标准

高质量蒸馏结果建议至少通过以下 8 项检查。

# 一、可触发性

应能明确回答：

- 什么时候该用这个 skill？
- 典型触发场景是什么？
- 用户会说什么样的话来触发它？

# 二、可执行性

应能明确回答：

- 这个 skill 不是纯概念介绍，而是真的能驱动 agent 做事吗？
- 它有输入、步骤、分支、输出格式吗？

# 三、可验证性

应能明确回答：

- 怎么知道 skill 用对了？
- 完成标准是什么？
- 哪一步失败算失败？

# 四、可复用性

应能明确回答：

- 这份内容是否已经抽掉过度依赖单一上下文的细节？
- 哪些是通用规则？哪些只是例子？

# 五、可追溯性

应能明确回答：

- 关键规则来自哪里？
- 来源版本是什么？
- 后续材料更新时怎么追踪？

# 六、边界清晰

应能明确写出：

- 不适用场景
- 高风险误用
- 需要用户确认的地方

# 七、压缩得当

应达到：

- 高信噪比
- 但没有丢失必要上下文

压太狠会变成口号；压不够又不能被稳定调用。

# 八、结构友好

应做到：

- 标题清晰
- 分块合理
- 结论前置
- 列表与短段落优先

这不仅适合人读，也更适合 agent/human 协作时快速扫描。

# 六、常见失败模式

# 一、只做摘要，不做操作化

结果是：

- 看起来很懂
- 实际上不能驱动 agent 行为

# 二、过度压缩，丢失边界条件

常见于：

- 书籍
- 深度文章
- 复杂教程

结果是：

- nuanced 观点被错误转成绝对规则

# 三、内容类型混淆

例如：

- 用 explanation 替代 how-to
- 把 reference 写成教程

结果是：

- agent 会在不该执行时执行
- 或在该执行时只会解释

# 四、来源污染

把：

- 不同版本
- 不同作者立场
- 不同情境约束

混成一个 skill，容易导致自相矛盾。

# 五、把暂时性决策误当长期原则

会议纪要与对话记录里最常见。

“这次这么做”不等于“以后都这么做”。

# 六、缺少反例和不适用条件

只写“该怎么做”，不写“何时别这么做”，会让 skill 非常危险。

# 七、没有验证闭环

skill 文本写得很好，但没有拿真实任务跑一遍，结果一上线就出现：

- 步骤顺序错误
- 依赖缺失
- 停止条件缺失
- 输出格式不稳定

# 八、维护断裂

技术文档和产品流程一旦变化，skill 却没有更新，就会很快失效。

# 七、对 `evo_skills` 最适合的蒸馏流程建议

结合当前项目的目标与约束，最适合 `evo_skills` 的不是外部 API 驱动的批处理蒸馏，而是：

**agent-only 的交互式分阶段蒸馏流程**。

推荐流程如下：

## 1. 先判断材料类型

当前 agent 在读原文后，先判断它更接近：

- tutorial
- reference
- explanation
- book-like framework
- notes / transcript

同时判断最终更适合生成：

- 操作型 skill
- 原则型 skill
- 教练型 skill
- 参考型 skill
- memory

## 2. 先做 source map，不急着总结全文

当前 agent 先产出：

- 内容概览
- 主题分块
- 重复点
- 高价值段落
- 明显噪音与可忽略部分

## 3. 提取四类原子单元

优先提取：

- Concepts
- Procedures
- Rules
- Constraints

如果原文还无法稳定落到这四类之一，就说明不该急着 build 正式 skill。

## 4. 重组成 `skill_spec.json`

当前 agent 将上一步结果重写为：

- `distillation`
- `examples`
- `skill_design`
- `mode_evaluation`
- `memory_design`
- `artifact_plan`

## 5. 设置 review gate

在调用 builder 之前，由用户至少确认：

- `display_name`
- `skill_id`
- `content_type`
- `supported_modes`
- `execution_level`
- `core_thesis`

## 6. 再调用本地脚本 build

builder 负责：

- 写 article package
- 写 child skill package
- 写 memory scaffold
- 更新 registry
- 刷新 summary

## 7. 最后做最小试跑

应让 agent 用新生成的 skill 处理 1 到 2 个最小任务，检查：

- 是否误触发
- 是否过度执行
- 是否遗漏边界
- 输出格式是否合理

# 八、针对不同材料的推荐蒸馏决策表

| 材料类型 | 推荐蒸馏重点 | 推荐产物类型 | 是否适合 execute |
|---|---|---|---|
| 文章 / 博客 | thesis、规则、边界、误用 | 原则型 / 教练型 skill | 通常否 |
| 书籍 | 框架、模型、跨章节重复原则 | 框架型 / 反思型 / 教练型 skill | 通常否 |
| 教程 / 操作指南 | 前置条件、步骤、验证、排错 | 操作型 skill | 通常是 |
| 技术文档 / API 文档 | 能力边界、参数、限制、任务包装 | 参考型 / 混合型 skill | 有条件 |
| 笔记 / 摘录 | 聚类、去重、反复出现的判断标准 | 轻量 skill / memory | 视情况 |
| 会议纪要 / 转录 | 决策、约束、禁忌、优先级 | memory / 约束型 skill | 通常否 |

# 九、对后续实现的建议

结合这次调研，后续在 `evo_skills` 中建议优先实现以下能力：

1. **材料类型判断**：在 distillation 开始前先分类。
2. **source map 阶段**：不要直接从原文跳到最终 skill。
3. **四类原子单元抽取**：Concepts / Procedures / Rules / Constraints。
4. **review gate 标准化**：build 前必须确认关键字段。
5. **最小试跑验证**：新 skill build 完后至少跑一个小样本任务。

这几项会比继续堆模板或继续增加脚本更重要，因为它们决定了 `evo_skills` 的内容质量上限。

# 十、结论

本次调研支持这样一个明确判断：

对于 `evo_skills` 这样的系统，最佳蒸馏过程并不是“让模型直接总结并写完全部文件”，而是：

**先按材料类型选择合适蒸馏路线，再由当前 agent 分阶段提炼出结构化 spec，再通过 review gate 与本地 build 把它转成正式 child skill。**

换句话说：

- 高质量蒸馏 = 分类 + 分层 + 操作化 + 验证
- 低质量蒸馏 = 直接摘要 + 直接落盘 + 缺乏边界与验证

这份报告应作为 `evo_skills` 后续继续实现 distillation 层与 routing 层的基础依据。
